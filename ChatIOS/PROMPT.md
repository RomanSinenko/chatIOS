Что есть сейчас:

- `ContentView.swift` как корневой экран с `NavigationStack`

Уже подключено:

Пока не реализованы:

- поиск пользователей через backend
- создание private chat из iOS

- WebSocket

# Что важно про backend
Backend уже существует и умеет базовый chat MVP.

Основные готовые HTTP endpoints:
- `POST /users/{user_name}` — создать пользователя
- `GET /users/search?query=...` — поиск пользователей
- `POST /private-chats/{user_id}/{peer_user_id}` — найти или создать private chat
- `GET /users/{user_id}/chats` — список чатов пользователя
- `GET /chats/{chat_id}?user_id=...` — мета-информация о чате
- `GET /chats/{chat_id}/messages` — история сообщений

WebSocket:
- `WS /ws/{user_id}`

Формат исходящего сообщения от клиента:

```json
{
  "chat_id": 1,
  "to_user_id": 2,
  "text": "hello"
}
```

Форматы входящих событий с сервера:
- `message`
- `system`
- `error`
- `message_ack`

Что уже поддерживается на backend:
- private chat flow
- self-chat
- список чатов
- история сообщений
- WebSocket realtime
- `message_ack`
- membership checks

# Что сделано в большом шаге `ios-auth-api-client`
Ветка: `ios-auth-api-client`

Сделано:
- создана модель `ChatUser`
- добавлен `Decodable` для разбора JSON от backend
- добавлен `CodingKeys`, чтобы связать backend поле `user_name` со Swift свойством `userName`
- создан `APIClient`
- добавлен базовый `baseURL` для локального backend `http://127.0.0.1:8000`
- реализован `createUser(userName:)`
- исправлена сборка URL через `appendingPathComponent`, чтобы не было двойного percent-encoding кириллицы
- `ContentView` теперь хранит `currentUser: ChatUser?`, а не только имя строкой
- `StartView` принимает `isLoading` и `errorMessage`
- при нажатии `Продолжить` iOS отправляет `POST /users/{user_name}`
- при успешном ответе backend пользователь сохраняется в `currentUser`
- после успешного создания пользователя открывается `ChatsListView`
- во время запроса кнопка показывает `Загрузка...`
- при ошибке показывается сообщение на стартовом экране
- для `400` показывается ошибка невалидного имени
- для `409` показывается ошибка занятого имени
- при недоступном backend показывается ошибка подключения

Текущий auth flow:

```text
StartView
  ввод имени
    -> APIClient.createUser(userName)
      -> POST /users/{user_name}
        -> ChatUser(id, userName)
          -> ContentView.currentUser
            -> ChatsListView
```

Пока auth временный через username. В будущем планируется регистрация/вход по номеру телефона.

# Формат нашей работы
Работаем так:
- сначала определяем крупный шаг, который нужно сделать
- под этот шаг создаём отдельную git-ветку
- дальше двигаемся короткими шагами
- на каждом коротком шаге AI пишет:
  - в какой файл нужно внести код
  - какую функцию, структуру или часть кода нужно добавить
  - сам код для вставки
- после кода AI отдельно объясняет:
  - как это работает
  - зачем это нужно
  - какие новые методы, функции или конструкции там использованы
- пользователь переписывает код вручную сам
- после того как пользователь переписал код и всё прочитал, он присылает `"+"`
- сообщение `"+"` означает: можно двигаться дальше к следующему короткому шагу

Важно:
- AI не должен торопиться и перескакивать через шаги
- AI не должен выдавать сразу большой объём реализации на много шагов вперёд
- один ответ = один понятный короткий шаг в рамках текущего большого шага

# Завершение большого шага
Когда большой шаг завершён:
- обновляем `PROMPT.md` по аналогии с backend-подходом
- фиксируем, что именно было сделано в рамках шага
- сразу определяем следующий большой шаг
- после этого делаем `git push`
- затем продолжаем работу уже от следующего большого шага

Это нужно, чтобы:
- сохранять живой контекст проекта
- не терять прогресс между этапами
- всегда понимать текущее состояние iOS-части
